# 카페 프로젝트 발표 대본

## 오프닝
안녕하세요. 오늘 소개해드릴 프로젝트는 카페 주문 웹앱 "카페"입니다.

## 왜 이 프로젝트를 만들었나
멋쟁이사자처럼 백엔드 스쿨 첫 번째 프로젝트로, 프론트엔드 기초(HTML, CSS, 자바스크립트)를 제대로 다지는 걸 목표로 잡았습니다. 그래서 프레임워크나 라이브러리에 기대지 않고, 누구나 한 번쯤 써본 "카페 주문"이라는 익숙한 주제를 골랐습니다. 익숙한 주제이다 보니 기능 설계에 드는 고민은 줄이고, 대신 화면 구조를 짜고 상태를 관리하는 프론트엔드 기본기와 저만의 디자인·차별화 기능을 만드는 데 더 집중할 수 있었습니다.

---

## 1. 기획 의도
- 이번 프로젝트는 카페 주문 웹앱을 처음부터 끝까지 직접 설계하고 만들어본 프로젝트입니다. 메뉴 탐색부터 옵션 커스터마이징, 장바구니, 주문, 마이페이지까지 실제 카페 서비스의 흐름을 바닐라 HTML, CSS, 자바스크립트로 구현했습니다.
- 디자인은 웜 뉴트럴 톤의 유럽풍 부티크 카페 컨셉으로 전체를 리디자인해서, 흔한 부트캠프 템플릿 느낌을 벗어나려고 했습니다. 모두 같은 카페 앱이라는 동일한 과제에서 출발했지만, 디자인과 세부 기능에서 다르게 가져가려고 한 부분입니다.
- 가장 신경 쓴 차별화 포인트는 **"나만의 레시피 저장 + 원클릭 재주문"** 기능입니다. 자주 마시는 커스텀 옵션 조합, 예를 들어 아이스, 라지, 두유 조합에 이름을 붙여 저장해두고, 상세 페이지에서 바로 불러와 재주문할 수 있게 만들었습니다. 다른 팀에서는 보기 힘든 기능이라고 생각합니다.

**핵심 한 줄**: "나만의 음료를 저장하고, 한 번에 다시 담는다"

---

## 2. 기술스택
- 프론트엔드는 HTML5, CSS3(Grid, Flexbox)와 순수 자바스크립트로만 만들었고, 별도 프레임워크는 쓰지 않았습니다.
- 데이터는 localStorage 기반으로 저장합니다. 장바구니, 즐겨찾기, 레시피, 주문내역을 이 방식으로 관리하고, 최근에는 회원가입/로그인 기능을 추가해서 계정별로 데이터가 분리되도록 만들었습니다.
- Git/GitHub으로 버전관리를 했고, 폰트는 Google Fonts의 Noto Sans/Serif KR을 사용했습니다.
- GitHub Pages로 실제 배포까지 완료했습니다. `https://kjy-ux.github.io/cafe/` 에서 바로 접속 가능합니다.

## 3. 시연 (라이브 데모 순서)
1. 랜딩 페이지 보여주기 — 부티크 컨셉 리디자인 강조, GitHub Pages 배포 링크로 접속
2. 메뉴 목록 → 상세 페이지 진입 — 옵션 커스터마이징(온도/사이즈/샷/우유/시럽 등) 보여주기, 담으면 오른쪽에 뜨는 장바구니 미리보기도 같이 보여주기
3. 옵션 조합을 "나만의 레시피"로 저장
4. 장바구니에 담고 주문 완료
5. 마이페이지 → "나만의 메뉴"에서 즐겨찾기 + 저장한 레시피 확인, 레시피 한 번에 재주문
6. 회원가입/로그인 진행 → 헤더에 이름/로그아웃 뜨는 것 확인, 로그아웃 후 다시 게스트 상태로 돌아가는 것도 보여주기
7. 관리자 페이지 접속 시도 → 로그인 가드로 막히는 것, 관리자 계정으로 로그인 시 들어가지는 것 보여주기

## 4. 문제해결 (트러블슈팅)
**① 레시피 패널 추가 후 카드가 중앙에서 밀림**
레시피 패널을 상세 페이지에 추가했더니, 원래 중앙 정렬돼 있던 메뉴 카드가 옆으로 밀리는 문제가 있었습니다. 패널을 별도로 `position: absolute`로 분리해서, 카드는 그대로 중앙 정렬을 유지하도록 해결했습니다.

**② GitHub Pages 배포 후 메뉴 이미지가 전부 깨짐**
로컬에서는 멀쩡했는데, GitHub Pages에 배포하고 나니 메뉴 이미지가 다 안 보였습니다. 이미지 경로를 절대경로(`/images/...`)로 써놨는데, GitHub Pages는 `깃허브아이디.github.io/저장소이름/`처럼 하위 경로에 배포되다 보니 경로가 어긋난 거였습니다. 스크립트 자기 자신의 위치를 기준으로 이미지 경로를 동적으로 계산하도록 고쳐서 해결했습니다.

**③ 로그인 도입 후 장바구니·즐겨찾기가 계정 구분 없이 전체 공용이었음**
나중에 로그인 기능을 추가하면서, 장바구니나 즐겨찾기가 계정 구분 없이 전체 공용으로 저장되는 문제가 있었습니다. localStorage 키를 로그인 계정 기준으로(비로그인 시에는 게스트로) 분리해서 해결했습니다.

**④ 관리자 페이지가 URL만 알면 누구나 접근 가능했음**
관리자 페이지가 로그인 없이도 URL만 알면 누구나 들어갈 수 있는 보안 허점이 있었습니다. 관리자 권한 체크 로직(requireAdmin)을 추가해서, 권한이 없으면 자동으로 홈으로 리다이렉트되도록 만들었습니다.

---

## 5. 개선점
- 지금은 localStorage 기반이라 비밀번호도 평문으로 저장되는 데모 수준입니다. 실제 서비스로 가려면 서버와 DB를 연동해서 제대로 된 인증과 암호화가 필요합니다.
- 결제 기능과 실시간 주문 상태 알림 기능도 다음 단계로 추가하고 싶습니다.
- 모바일 반응형도 지금 어느 정도 대응은 되어 있지만, 터치 인터랙션 같은 세부 UX는 더 다듬을 여지가 있습니다.
- 메뉴 이미지도 지금은 수동으로 점검했는데, 앞으로는 등록 시점에 품질을 자동으로 체크하는 프로세스가 있으면 좋겠습니다.

## 예상 질문 대비
**Q. DB(데이터베이스) 연동은 안 했나요?**
A. 이번 프로젝트는 프론트엔드와 localStorage로 구현했습니다. 다만 향후 확장을 대비해서 Supabase에 회원, 메뉴, 주문, 장바구니, 즐겨찾기, 레시피까지 동일한 구조로 데이터 모델(스키마)을 미리 설계해뒀습니다. 실제 서비스에 연동하는 건 다음 단계 목표입니다.

**Q. 왜 프레임워크를 안 썼나요?**
A. 기초를 제대로 익히는 게 이번 프로젝트의 목적이라, 바닐라 HTML/CSS/JS로 직접 DOM을 다루면서 만들었습니다.

**Q. 로그인 기능은 안전한가요?**
A. 지금은 데모 수준으로, 비밀번호를 평문으로 저장하고 있어서 실제 서비스에는 쓸 수 없는 구조입니다. 실제 배포를 한다면 서버 인증과 암호화가 반드시 필요합니다.
